# Anthropic Compatibility Upgrade Plan

## Purpose

This document defines the upgrade plan for the `POST /v1/messages` and
`POST /v1/messages/count_tokens` compatibility layer in `codex2gpt`.

The immediate target is not "full Anthropic API parity". The target is a
stable, practical compatibility layer that allows `free-code` /
`claude-sh4wnsu` style Anthropic clients to use the local Codex-backed proxy
without protocol-level breakage on normal agent workflows.

## Background

`codex2gpt` already exposes an Anthropic-compatible surface:

- `POST /v1/messages`
- `POST /v1/messages/count_tokens`
- named SSE events for streaming responses
- Claude model aliases such as `claude-opus-4-6`, `claude-sonnet-4-6`,
  and `claude-haiku-4-5`

However, the current implementation is still a baseline shim. It supports the
simple request/response shape needed for basic text prompts, tool use, and
light reasoning, but it does not yet cover the request shape generated by
modern Claude Code style clients such as `free-code`.

Those clients send additional body fields, beta headers, richer tool schemas,
more content block types, and a real `countTokens` workload that goes beyond
the current estimator.

## Goal

Deliver a Claude-Code-friendly Anthropic compatibility layer for local proxy
use while preserving the stability of the existing:

- OpenAI Chat endpoint
- native Responses endpoint
- Gemini-compatible endpoint

## Non-Goals

The first implementation pass should **not** attempt to provide:

- full Anthropic API parity
- full server-side tool semantics
- remote web search / advisor / internal Anthropic-only features
- full fidelity support for every beta header
- silent passthrough of unsupported blocks with undefined semantics

If a feature cannot be represented safely on top of the current Codex-backed
Responses flow, the proxy should either:

- degrade explicitly and safely, or
- reject the request with a clear Anthropic-style error

## Compatibility Strategy

## 1. Separate Anthropic normalization from generic payload stripping

The current generic unsupported-field stripping is too blunt for Anthropic
traffic. Several fields that are currently removed at the generic layer are
meaningful to the Anthropic client and should be translated or explicitly
ignored only inside the Anthropic code path.

The implementation should introduce an Anthropic-specific normalization stage:

1. parse request
2. validate Anthropic request shape
3. normalize Anthropic-only fields into an internal representation
4. translate that internal representation into the Codex Responses payload
5. drop only fields that are confirmed to be non-forwardable

This keeps the Anthropic path understandable and reduces accidental breakage of
OpenAI and Gemini paths.

## 2. Classify Anthropic fields into three buckets

### Bucket A: must translate in the first pass

| Field or feature | Target handling |
| --- | --- |
| `model` | keep current alias mapping |
| `system` | map to proxy instructions |
| `messages[].content.text` | map to Responses text items |
| `messages[].content.tool_use` | map to function calls |
| `messages[].content.tool_result` | map to function call outputs |
| `thinking.budget_tokens` | map to best-effort Responses reasoning |
| `thinking.type = adaptive` | map to best-effort Responses reasoning without failing |
| `output_config.format` | map to Responses structured output config |
| `output_config.effort` | map to Responses reasoning effort |
| `tools` | keep current tool mapping |
| `tool_choice` | keep current tool choice mapping |
| `image` blocks | map to Responses image input |

### Bucket B: accept and ignore safely in the first pass

| Field or feature | First-pass behavior |
| --- | --- |
| `metadata` | accept, do not forward upstream |
| `context_management` | accept, do not forward upstream |
| `output_config.task_budget` | accept, do not forward upstream |
| `speed` | accept, do not forward upstream |
| `anthropic-beta` header | accept, do not error on it |
| tool schema `cache_control` | accept, do not forward upstream |
| tool schema `defer_loading` | accept, do not forward upstream |
| tool schema `eager_input_streaming` | accept, do not forward upstream |

### Bucket C: explicit reject in the first pass

| Field or feature | First-pass behavior |
| --- | --- |
| `document` blocks | reject with `invalid_request_error` unless a safe mapping is implemented |
| `tool_reference` blocks | reject with `invalid_request_error` unless a safe mapping is implemented |
| `server_tool_use`-style semantics | out of scope for the first pass |
| opaque beta-only server content | reject unless it can be translated safely |

This is intentionally stricter than silent dropping. For unsupported semantic
content, explicit failure is safer than creating subtly corrupted conversations.

## Translation Design

## 1. Top-level request handling

### `model`

Keep the current alias-based model resolution. Continue to expose local model
aliases such as:

- `claude-opus-4-6`
- `claude-sonnet-4-6`
- `claude-haiku-4-5`

The alias layer remains the compatibility contract. The proxy is not expected
to accept arbitrary Claude model IDs in the first pass.

### `max_tokens`

Continue to require `max_tokens` for Anthropic requests. This preserves client
expectations and keeps the endpoint shape consistent.

The raw value should remain a validation input and compatibility input. It
should not be blindly forwarded into the upstream Codex Responses payload.

### `thinking`

Current behavior maps `budget_tokens` into coarse reasoning effort bands. The
upgrade should make this more explicit:

- `thinking.type = adaptive`
  - normalize into an internal thinking mode
  - translate to a best-effort Responses reasoning setting
  - do not reject only because adaptive is requested
- `thinking.type = enabled` with `budget_tokens`
  - map budget ranges into reasoning effort bands
- missing or disabled thinking
  - leave reasoning unset or at default behavior

This remains a semantic approximation, not a promise of full Anthropic
thinking parity.

### `output_config`

Support two useful sub-fields in the first pass:

- `output_config.format`
  - translate into the same structured output representation already used on
    the OpenAI-compatible path
- `output_config.effort`
  - translate into Responses reasoning effort if the value is compatible

Accept but do not forward:

- `output_config.task_budget`

### `metadata`, `context_management`, `speed`

These should be accepted to avoid client-side protocol failure, but ignored for
the first pass. The proxy may log them in debug mode for observability.

## 2. Content block handling

### `text`

Keep current behavior.

### `tool_use`

Keep current behavior:

- assistant-only
- must contain `id`
- must contain `name`
- `input` must be an object

### `tool_result`

Keep current base behavior:

- user-only
- must contain `tool_use_id`
- convert text content to function call output

The first pass should continue to support:

- string content
- list of text blocks

It does not need to support tool result media nesting in the first pass.

### `thinking` and `redacted_thinking`

Keep the current history behavior:

- allowed only in assistant messages
- ignored when converting request history into Responses input

This is the safest current behavior because previous reasoning blocks do not
map cleanly onto Responses input items.

### `image`

Add support for user image input.

Target mapping:

- Anthropic `image` block
- normalized into a Responses `input_image` item

The implementation should support the most common image payload forms used by
Anthropic SDK clients and explicitly reject malformed image blocks.

### `document`

Do not silently drop document blocks.

Until the proxy has a safe file/document mapping into the upstream model
surface, it should return a clear `invalid_request_error` describing that
document blocks are not yet supported by the local Anthropic compatibility
layer.

### `tool_reference`

Do not silently drop tool references.

`tool_reference` is not just a transport detail. It represents tool-search
semantics and deferred tool materialization. Unless the proxy adds a deliberate
translation strategy, it should reject these blocks explicitly.

## 3. Tool schema handling

The Anthropic compatibility layer should keep accepting normal tool definitions
and preserve the current mapping:

- `name`
- `description`
- `input_schema`

For first-pass compatibility, the following extra tool fields should be
accepted and ignored:

- `cache_control`
- `defer_loading`
- `eager_input_streaming`

This avoids request failure from modern clients while keeping the upstream
translation conservative.

## Response Design

## 1. Non-streaming response

Keep the current response envelope:

- `id`
- `type = message`
- `role = assistant`
- `model = requested Anthropic model`
- `content`
- `stop_reason`
- `stop_sequence`
- `usage`

Keep the current mapping for:

- assistant text -> Anthropic `text` content block
- function calls -> Anthropic `tool_use` content block
- cached input tokens -> `cache_read_input_tokens`

## 2. Streaming response

Keep the current named Anthropic SSE event stream as the baseline.

Required first-pass guarantees:

- text output streams as `text_delta`
- tool call arguments stream as `input_json_delta`
- thinking output streams as `thinking_delta`
- final stream always closes with `message_delta` and `message_stop`

`signature_delta` is optional and should not block the first pass.

## 3. Buffered compatibility path

Keep the current buffered fallback for Claude-Code-like clients that present as:

- Bearer-authenticated Anthropic clients, or
- `claude-code` user agent clients

This path is important because some Anthropic SDK clients are more stable when
they receive a preassembled SSE response rather than a directly relayed live
event stream.

## Count Tokens Strategy

The current `count_tokens` endpoint is too rough for real Anthropic clients.
It only estimates the translated Responses payload and does not account for the
actual Anthropic request shape in enough detail.

The upgraded estimator should include:

- `system`
- all message text
- tool call names and argument JSON
- tool result text
- tool schema names, descriptions, and input schemas
- image blocks with a fixed heuristic overhead
- structured output schema overhead

This remains an estimate, not a true upstream count, but it should be much
closer to the client's real expectation than the current implementation.

## Error Handling Rules

The Anthropic path should follow these rules:

- preserve Anthropic-style error payloads
- reject malformed or semantically unsupported content explicitly
- never silently discard semantic content except:
  - assistant history `thinking`
  - assistant history `redacted_thinking`
- when rejecting unsupported features, name the block or field directly in the
  error message

Examples of acceptable first-pass explicit errors:

- `unsupported user content block type: document`
- `unsupported tool_result content block type: image`
- `unsupported user content block type: tool_reference`

## Acceptance Criteria

The upgrade is complete for the first pass when all of the following are true:

1. A `free-code`-style Anthropic request with normal text messages succeeds.
2. A request with tool use and tool results succeeds.
3. A request with `thinking.type = adaptive` does not fail at the proxy layer.
4. A request with `output_config.format` produces a valid translated request.
5. A request with `output_config.effort` influences the translated reasoning
   setting.
6. A request with user image input succeeds.
7. `POST /v1/messages/count_tokens` includes tool overhead and media overhead.
8. `document` and `tool_reference` are rejected explicitly rather than being
   dropped silently.
9. Existing OpenAI and Gemini compatibility tests still pass.

## Recommended Implementation Boundary

The first pass should stop after the proxy can reliably support:

- normal Claude-Code-style local chat
- tool use
- thinking
- structured outputs
- image input
- better token estimation

It should deliberately postpone:

- document ingestion
- tool reference semantics
- full beta header semantics
- server tool flows
- connector-text-specific features

That boundary keeps the project pragmatic, testable, and aligned with the real
short-term goal: making local Anthropic-compatible agent tooling work well on
top of the Codex-backed proxy.
